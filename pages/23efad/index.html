凤凰项目：一个IT运维的传奇故事

# 作者简介：

吉恩·金（Gene Kim）

信息技术流程研究所联合创始人、研究总监，Tripwire 公司创始人，担任公司 CTO 长达 13 年之久，一直热衷于研究如何提高 IT 组织的效能。

凯文·贝尔（Kevin Behr）

PraxisFlow 咨询公司首席科学官，信息技术流程研究所联合创始人，拥有 25 年以上的 IT 管理经验，常为 CEO、CIO、CTO 等提供指导和建议。

乔治·斯帕福德（George Spafford）

高德纳公司高级研究总监，行业分析师，在 IT 运维方面拥有丰富经验，曾在多个国家提供过信息技术治理和流程改进等方面的咨询和培训。

# 书本简介

本书讲述了一位 IT 经理临危受命，在未来董事的帮助和自己“三步工作法”理念的支撑下，最终挽救了一家具有悠久历史的汽车配件制造商的故事。小说揭示了管理现代 IT 组织与管理传统工厂的共通之处，让读者不仅能对如何管理 IT 组织心领神会，更重要的是将以完全不同于以往的视角来看待自己的工作环境。

# ❓ 如何用一句话介绍这本书?

《凤凰项目：一个 IT 运维的传奇故事》一书一共有 340 页，其中有 172 页用来描述 IT 的困境（第一部分），120 页来描述一些小的改进（第二部分），最后 50 页才上干货（第三部分）。

# ⚡ 重要章节/内容摘录

# 第一部分（一地鸡毛）

  • 面对不可能的任务，虽然知道应该尽可能跟长官据理力争，但有时候却不得不说一句”Yes，sir！“，然后去攻下某座山头。

  • 付薪系统危机，导致公司无法下发薪水，面临公会的压力。变更不透明，导致运维瞎忙活影响了 SND（Storage Area Network）导致更大范围的影响，最终定位到是 John 团队下的成员由于要休假，积极提交的变更没有经过测试后就发布。Urgent request + 休假，忙中出错的典型案例。

  • 无法完成的任务！无法延期的凤凰项目，千疮百孔的安全审计，堆积如山的 hot-fix！

周三，负责零售业务的高级副总裁 Sarah 召开会议，表示凤凰项目有延期的风险。大家纷纷表示这是一定是 Ops（主人公的部门）的锅。

周四，审计部门告诉 CEO 他们找到了 900 多条有关 IT 系统安全的问题，必须在外部审计组入驻前解决掉。这个任务，自然又是落在主人公头上。

周五，主人公发现整个部门都在忙 hot-fix，压根没有时间去解决他们头上的“两座大山”——凤凰项目和审计。

# 错综复杂内部管理：

  • a) 遇到瓶颈——组员 Brent 似乎是所有项目的关键人物，这太危险了。最后只能下令让 Brent 专注于凤凰项目，并命令 SAN 组在没有 Brent 的情况下修复系统。

  • b) 似乎永远在做 hot-fix，没有时间去做那些重要的事情，比如凤凰项目。

  • c) 产生问题 b 的根源在于交流不畅（we don’t talk to each other），不知道其余组干了什么，因此在部署时常常发现有冲突。交流不畅是因为大家对会议没有信心，认为会议冗长而无效。

  • d) 问题 b 和问题 c 有时又是矛盾的，比如 Brent 要修复其它系统，没时间参加会议去讨论凤凰项目的 config，他不参加会议就意味着未来会出现 bug，而他则要花时间去修复 bug，那么他就更没时间参加会议了。

  • e) 管理混乱：不知道我们部门向其它部门承诺了哪些活，不知道这些活的优先级，也不知道大家都在忙什么（WIP & resource availability）。

# 针对现状的改进：

1、明确变更定义：‘变更’就是对应用程序、数据库、操作系统、网络或硬件进行的物理、逻辑或虚拟操作，并且这样的操作可能对相关服务产生影响。

2、定位约束点：Brent

3、减少 WIP：限制每个人同时只能做一件事情，这样就能避免大家同时做 hot-fix，而忽略了凤凰项目。

有关 Brent 的故事

1. 灵活机动 vs 不稳定因子： 如果市场部门需要一个新功能，而你发现这个功能只需要稍加改动，十分钟之内就能搞定。你会怎么做？直接改还是根据流程提交 CR，测试，部署并记录？Brent 会选择直接做，结果就是信用卡支付系统宕掉时，大家都不知道怎么办，因为没人知道 Brent 做了什么改动，虽然最后 Brent 修复了问题，但主人公已经把他认定为一个不稳定因子。

2. 热心助人 vs 无法专注于重要的项目： 尽管 Brent 的唯一任务就是凤凰项目，但常常有人需要“一点点帮助”，这个帮助可能只有 20 分钟，但是这样一个个“一点点帮助”加起来，导致的就是 Brent 一直在做这些零碎的事情，没有真正的产出

3. 不懂得如何拒绝： 如果别人只要“帮一点点小忙”，实在是难以拒绝的。而如果你决绝了，对方可能会直接抛出某个副总裁的名字，那 Brent 只好屈服了。

4. 从不记录： 导致除了他以外没有人熟悉公司的系统
