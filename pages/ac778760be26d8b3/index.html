# Nginx网关

网关一般都是作为一个网站流量的总入口，负责对用户请求的分发处理。除了nginx软网关，还有硬网关，比如F5、Netscaler等。这里我们主要讲一下nginx网关。

# 0.前置条件

在我们的生产环境中，nginx一般都是部署在linux服务器上的，所以有一些前置的系统调优是必不可少的。

以下是一些常见的优化步骤：

    1. 调整文件描述符限制：Nginx使用大量文件描述符，可以通过增加文件描述符限制来提高性能。可以通过修改/etc/security/limits.conf文件来增加文件描述符限制。
    1. 调整内核参数：根据服务器硬件和网络环境，可以调整一些内核参数以优化性能。例如，可以调整TCP连接参数、内存分配参数等。可以通过修改/etc/sysctl.conf文件来设置内核参数。
    1. 使用高性能文件系统：选择适合服务器的高性能文件系统，如XFS或ext4，并使用适当的选项进行挂载。

调整内核参数： 内核参数是控制操作系统内核行为的设置。通过调整内核参数，可以优化系统的网络和内存管理，提高性能和稳定性。主要包含TCP连接参数、内存管理参数、网络参数、文件系统参数等。

调整文件描述符限制： 文件描述符是操作系统用于管理打开文件的标识符。Nginx使用大量的文件描述符来处理并发连接和文件操作。如果文件描述符限制过低，可能导致Nginx无法处理更多的连接或打开更多的文件，从而影响性能和稳定性。

# 1.软件安装

这里我们不会参考文档一步步说，核心的还是关注一下nginx在linux环境所需要做的事情。

1、先从官网下载nginx的源码包，然后解压到指定目录。tar -zxvf nginx-x.x.x.tar.gz 2、进入解压后的目录，执行./configure命令，这个命令会检查当前环境是否满足nginx的安装条件，如果不满足，会提示缺少哪些依赖包，需要自己去安装。在这个环境可以配置软件安装的目录，以及软件运行所需要的module。 3、执行make命令，编译源码。它会根据配置选项编译Nginx的可执行文件和模块。 4、执行make install命令，安装Nginx。它会将编译好的Nginx可执行文件和模块复制到指定目录。

关键点：

  • configure脚本根据系统环境和配置选项生成Makefile，用于编译Nginx。
  • make命令根据Makefile执行编译过程，生成可执行文件和模块。
  • make install命令将编译好的文件安装到系统中。

与windows不同，linux下的软件安装一般都是源码安装，所以需要自己去解决依赖问题，这个时候就需要我们去安装一些依赖包，比如openssl、pcre、zlib等。

# 2.常用配置

这里讲一下我遇到的生产环境中常用的配置。

# 2.1. ip黑名单过滤

项目上线初期，经常会有一些恶意攻击，比如爬虫、恶意请求等，这些请求会占用大量的服务器资源，导致服务器无法正常对外提供服务。所以我们需要对这些恶意请求进行过滤，这里我们使用nginx的ip黑名单功能。

首先使用map指令定义$blocked变量来对需要过滤的ip进行标识，并把需要过滤的ip写入到deny.conf文件中。

$http_x_forwarded_for是一个包含客户端IP地址的请求头，通常用于代理服务器转发请求时传递客户端的真实IP地址。

include deny.conf;指令用于引入名为deny.conf的配置文件。这个配置文件中包含了一些规则，用于根据$http_x_forwarded_for的值来决定是否将$blocked变量设置为非零值。

map $http_x_forwarded_for $blocked {
    include deny.conf;
}

2
3

最终，根据$http_x_forwarded_for的值和deny.conf中的规则，$blocked变量将被设置为相应的值。如果匹配到了规则，$blocked将被设置为非零值；否则，它将保持默认值0。

👌，接下来就只需要去到server中进行判断即可。

if ($blocked) {
    return 403;
}

# 2.2. 解决小程序静态资源跨域问题

在小程序中，针对访问的网络资源，小程序会对域名进行校验，如果不在白名单中，就会报错。所以我们需要在nginx中配置一下跨域问题，将一些外部的网络资源通过proxy_pass转发到对应的域名下。

# 2.3. 有状态服务session保持问题

在一些有状态服务中，会将用户的登录信息保存在session中，但是通过nginx转发的时候，如果只单纯的配置upstream，那么每次请求都会转发到不同的服务器上，这样就会导致session信息丢失，所以我们需要在nginx中配置一下session保持。


 




upstream app_servers {
  ip_hash;
  server 192.168.1.101;
  server 192.168.1.102;
}

ip_hash指令的作用是按照客户端IP的hash结果分发请求,这样来自同一IP的请求会被代理到同一台后端服务器，由于session信息存储在后端服务器上,这样可以使同一客户端的请求始终路由到同一台服务器。

当然这里面也有一些第三方session存储方案，比如redis、memcache等，这里就不做过多的介绍了。

# 2.4. 故障问题临时导流

在生产环境中，我们有可能会遇到一些故障，有些可能影响用户体验，但有一些可能会对服务器造成严重的影响，这个时候我们就需要对这些故障进行临时导流，避免影响到大面积的用户。

例如：/getBill接口，由于数据量太大，没有做好统计优化导致大量的查询sql阻塞，导致整个服务的数据库链接池被占满，这个时候我们就需要对这个接口进行临时导流，避免影响到其他接口。

location /getBill {
    return 503;
}

# 2.5. 80/443端口

在linux中，1024以下的端口是需要root权限才能使用的，所以在生产环境时并不能直接使用80/443端口，这个时候我们可能就需要上层的负载均衡来进行端口转发。
